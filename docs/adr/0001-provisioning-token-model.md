# ADR 0001 — Provisioning Token Security Model

**Status:** Accepted  
**Sprint:** 17 (HERMES-II)  
**Date:** 2026-06-28

## Context

Atlas Runtime Agents need to self-register with the platform without
pre-existing credentials. A provisioning token acts as the bootstrap
credential that authorises one or more registrations scoped to a company.

Key requirements:
1. Tokens must never be recoverable if the database is compromised.
2. Lookups must be O(1) without iterating all tokens.
3. The model must support revocation and expiry.

## Decision

**Hash algorithm: SHA-256 (deterministic)**

| Approach | Pros | Cons |
|---|---|---|
| bcrypt | Adaptive cost; password-safe | Non-deterministic; can't look up by hash |
| SHA-256 | Deterministic; fast; O(1) lookup | Not adaptive — unsuitable for passwords |
| HMAC-SHA-256 | Adds server-side secret | Extra secret management overhead |

Provisioning tokens are 256-bit random values generated by
`crypto.randomBytes(32)`. For random secrets at this entropy level,
SHA-256 without a salt is cryptographically sound — brute-force and
rainbow-table attacks are infeasible.

**Token format: `slp_<hex(32 bytes)>`**

- `slp_` prefix: human-recognisable namespace ("Seltriva Letsgo Provisioning")
- 64 hex chars: 256 bits of entropy
- Total length: 68 chars

**Prefix index: first 12 chars of raw token**

Allows fast row lookup (`WHERE tokenPrefix = ?`) before the hash comparison,
avoiding a full-table scan on the `tokenHash` column in high-volume scenarios.

**Raw token delivery**

The `ProvisioningToken.create()` factory returns `{ token, rawToken }`.
`rawToken` is returned to the caller exactly once and is not stored anywhere
in the system. The persistence layer only sees `tokenHash`.

## Consequences

- Token lookup is O(1): `WHERE tokenHash = sha256(rawToken)`.
- Database breach does not expose any usable token.
- Token recovery is impossible (intentional).
- `tokenPrefix` adds a secondary index at the cost of 12 bytes per row.
- Future migration to HMAC-SHA-256 is possible by adding a `hashVersion`
  column and re-issuing tokens with a rolling rotation.
